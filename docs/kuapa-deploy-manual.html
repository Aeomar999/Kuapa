Architecture at a glance

Both clients talk to one Railway service over HTTPS (/api/v1) and WSS (socket.io). The server reaches Neon Postgres through Prisma 7's pg driver adapter. External services are wired purely through environment variables.

topology
  Mobile APK ─┐                        ┌─ Cloudinary · Paystack · Sentry
  (Expo/EAS)  ├─ HTTPS /api/v1 ──▶  Railway  ├─ PostHog · Arkesel SMS · SMTP
  Admin (Next)┘   WSS socket.io    NestJS API ┴─ Google Maps · Better Auth Cloud
                                      │  node dist/main · binds 0.0.0.0:$PORT
                                      ▼  Prisma 7 (pg adapter) · DATABASE_URL (SSL)
                                  Neon Postgres

1.4 — done: bundle renamed to com.kuapa.app

Store identity renamed from com.bexiemart.app → com.kuapa.app across every functional file: app.json (bundleIdentifier + package), the mobile auth client (Origin header + expo scheme/storagePrefix → kuapa), server trustedOrigins (added com.kuapa.app://, removed the dead bexiemart origins), and the Android assetlinks.json + iOS apple-app-site-association.

!
One follow-up after your first EAS buildA renamed Android package gets a new signing keystore, so the sha256_cert_fingerprints in assetlinks.json is now stale. Grab the new one with eas credentials (Android → production), replace it, and redeploy the server so App Links auto-verify. The iOS AASA still has a TEAMID. placeholder — swap in your Apple Team ID.
!
Two stale strings left on purposeeas.json iOS appleId: developer@bexiemart.com (placeholder for real Apple credentials) and app.json Sentry project: bexiemart-mobile (must match the real project in Sentry org kreddible — renaming breaks source-map uploads).

1.5 — required env not in .env.example

better-auth.ts uses process.env.BETTER_AUTH_API_KEY! (non-null) for the dash + sentinel plugins. It's missing from .env.example. Get a key from Better Auth Cloud and set it on Railway, or the server throws at boot.

A

Database on Neon

Create the project

Neon console → New Project. Name kuapa. Pick a region close to your Railway region (keep them together to cut latency). Postgres 16+, default database neondb.

Grab the connection string

Neon offers direct (ep-xxxx…) and pooled (…-pooler…) endpoints.

i
Use the direct endpointThe NestJS server is a single long-lived process that keeps its own pg pool, so the direct connection avoids PgBouncer prepared-statement quirks and one URL serves both runtime and migrations. Switch runtime to pooled only if you later run many replicas. Keep ?sslmode=require — Neon mandates TLS.
DATABASE_URL shape
postgresql://<user>:<pw>@ep-xxxx.<region>.aws.neon.tech/neondb?sslmode=require&channel_binding=require

Run the initial migration — from your machine

Migrations aren't part of the Railway build, so apply the repo's migrations (0_init, support tickets, escrow, super-admin, banners) against Neon once.

PowerShell · apps/server
cd "C:\Users\Jerry\Desktop\PROJECT 2026\Kuapa\Kuapa\apps\server"
$env:DATABASE_URL = "postgresql://…neon.tech/neondb?sslmode=require&channel_binding=require"
npm install --legacy-peer-deps
npm run prisma:generate
npm run prisma:deploy   # = prisma migrate deploy → applies all migrations
npm run prisma:studio   # verify tables exist
!
Free-tier autosuspendNeon suspends compute after ~5 min idle; the first request after idle cold-starts (a few hundred ms). Railway's health check pinging the DB keeps it warm-ish. Upgrade the plan to disable autosuspend for production.

Backend API on Railway

Create the service

Railway → New Project → Deploy from GitHub repo → your kuapa repo. Leave Root Directory as /: the root package.json scripts drive the build (npm --prefix apps/server install … && prisma generate && nest build) and start (node dist/main). Under Settings → Networking, Generate Domain → note it (e.g. kuapa-production.up.railway.app).

Set the environment variables

✕
Why NPM_CONFIG_PRODUCTION=false is mandatoryNODE_ENV=production makes npm skip devDependencies — but the build needs @nestjs/cli (nest build) and prisma (prisma generate), both devDeps. Without the flag the build fails with nest: not found / prisma: not found. Setting both keeps prod runtime behavior and a working build.

Generate the auth secret

BETTER_AUTH_SECRET

# Bash

Admin portal on Vercel

Standalone Next.js 16 app in apps/admin; it talks to the API purely over HTTP/WS via two public vars.

Import & configure

  1. Vercel → Add New → Project → import the GitHub repo.
  2. Root Directory: apps/admin (Edit → select — it's a monorepo).
  3. Framework auto-detects Next.js.
Vercel · Environment Variables
NEXT_PUBLIC_API_URL=https://kuapa-production.up.railway.app/api/v1   # includes /api/v1
NEXT_PUBLIC_WS_URL=https://kuapa-production.up.railway.app           # bare origin
✕
Wire the admin URL back into the APIWhatever domain Vercel assigns must be in the server's CORS_ORIGIN and Better Auth trustedOrigins or login fails. kuapa-admin.vercel.app, admin.kuapa.com, and kuapa.com are already in the code defaults. A custom domain means: add to CORS_ORIGIN (redeploy API) + add to trustedOrigins in better-auth.ts (commit, redeploy).

Log in at the Vercel URL with the seeded ADMIN_EMAIL / ADMIN_PASSWORD. The account is pre-verified and super-admin, so you can manage the admin team from inside the portal.

i
Public vs secretEXPO_PUBLIC_* vars ship inside the app bundle — only ever public keys (API URL, Paystack public key, Cloudinary cloud name, PostHog key). Real secrets live only on Railway. The lone build-time secret is SENTRY_AUTH_TOKEN (source-map upload).

Build the APK

eas build
eas build --platform android --profile preview      # shareable internal APK
eas build --platform android --profile production   # auto-incremented release APK
eas build:list --platform android --limit 5         # get the download URL

EAS builds on their servers (no local Android SDK). Download the .apk, install on a device (enable "Install unknown apps"). It reaches Railway over HTTPS. Smoke-test: register/login (watch railway logs), email-verify deep link opens the app via kuapa://verify-email, image upload, Paystack, maps.

i
OTA updatesAfter an install, ship JS-only changes without a new APK: eas update --branch production -m "…". Native changes (new packages, permissions, app.json native config) still need a fresh eas build.

Production hardening

  • Rotate BETTER_AUTH_SECRET, SMTP password, and any key that ever sat in a committed file.
  • Switch Paystack and other sandbox integrations to live keys.
  • Restrict the server Maps key by API + IP; restrict native keys by package/bundle + SHA-1.
  • Disable Neon autosuspend (or accept cold starts); enable Neon PITR/backups.
  • Add custom domains — api.kuapa.com (Railway), admin.kuapa.com (Vercel) — and update BETTER_AUTH_URL, CORS_ORIGIN, trustedOrigins, client env, and the App Links host.
  • Tune Sentry sample rates; confirm both server + mobile DSNs report.
  • Add a second replica only after moving DATABASE_URL to Neon's pooled endpoint.
  • Take a Neon branch for staging before testing risky migrations.
  • Finalize store identity (bundleIdentifier/package) before publishing — changing it later means a new listing.
